## 1\. 项目背景

Zenith Pay Platform 是一款从零全新自研、企业级标准的云原生聚合支付微服务平台。项目仅参考开源项目 AgPayPlus 成熟的支付业务场景与领域设计理念，覆盖商户管理、多渠道支付交易、退款处理、代理分润、资金结算、异步交易通知、交易对账清算等支付全链路核心能力。

**本项目绝非旧项目迁移、版本升级或二次改造**。彻底摒弃传统三层架构、数据表驱动开发、业务高度耦合、架构零散无规范的老旧系统缺陷。基于真实金融支付业务规则，全程依托 **\.NET 10 最新技术体系从零落地**，完整实现标准化 DDD 领域建模、微服务领域边界拆分、事件驱动架构、云原生容器化部署与全自动 DevOps 体系，打造一套无技术债、可商用、可横向扩展、规范标准化的现代企业级支付中台架构标杆。

## 2\. 项目核心建设目标

### 2\.1 业务目标：基于真实支付场景标准化 DDD 领域建模

支付系统具备业务规则复杂、订单生命周期长、状态机流转频繁、多渠道并行接入、资金链路严谨、异步交互多、一致性要求极高的业务特征。传统以数据库表为中心的开发模式，会导致业务逻辑散乱、领域边界模糊、资金规则不可控，完全无法适配金融级支付系统要求。

本项目全程采用**领域驱动设计（DDD）**思想，**以业务领域为设计核心，而非数据表驱动**，通过「实体、值对象、聚合根、领域服务、领域事件、领域规约」完成标准化业务建模，让代码结构完全贴合支付业务本质。

核心支付领域分层模型：Payment\.Domain 包含支付订单、交易流水、支付渠道、支付聚合根、订单状态机、全套支付领域事件等核心业务定义，严格分层隔离，保证资金业务高内聚、高一致、高可维护。

### 2\.2 架构目标：落地生产级云原生微服务架构

摒弃传统单体与伪微服务架构，落地标准**云原生微服务架构**，实现流量统一管控、安全统一拦截、服务独立治理、业务事件解耦、容器化弹性部署，完全适配企业生产环境。

**整体架构请求拓扑（最终定稿）**

客户端请求 → YARP 网关（统一入口、公钥验签、无状态认证、流量治理）→ 四大业务微服务集群 → RabbitMQ 事件总线 → 异步业务消费与任务调度

## 3\. 企业级技术基础设施底座（生产级落地）

本项目采用**行业标准生产级安全架构**：Identity 认证中心只负责「私钥颁发令牌」，**所有请求鉴权、验签、权限拦截全部在 YARP 网关层无状态完成**，业务服务不处理任何认证逻辑，无需远程调用认证中心，请求链路性能更高、架构更安全、彻底解耦。

### 3\.1 IdentityServiceCenter 统一身份认证中心

- **技术栈**：OpenIddict

- **核心定位**：独立静态令牌颁发中心，**仅签发、不校验**

- **核心能力**：基于私钥签发 JWT 令牌、OAuth2/OpenID Connect 协议支持、客户端管理、角色权限配置、全局密钥统一管理

### 3\.2 ApiGateway（YARP 高性能网关）

- **技术栈**：YARP \.NET 官方高性能反向代理网关

- **核心定位**：系统唯一流量入口，承载全链路安全与流量治理

- **生产级核心能力**：服务路由转发、**JWT 公钥本地验签、无状态安全认证、接口参数验签、密钥兼容解耦**、流量限流、服务聚合、请求拦截、日志审计

### 3\.3 Consul 服务治理中心

- **核心能力**：微服务自动注册、动态服务发现、健康检查、故障自动剔除、分布式 KV 全局配置热更新

- **架构价值**：实现服务动态扩容、无感知上下线、统一配置管理，保障分布式集群高可用

### 3\.4 SchedulerJob 分布式任务调度

- **技术栈**：Quartz\.NET

- **核心能力**：超时订单关闭、支付状态异步同步、自动对账、异常任务重试、周期性资金清算补偿

### 3\.5 自研 Zenith 系列通用 NuGet 组件

全部封装为独立 NuGet 包全局复用，统一技术规范、避免重复造轮子：

- Zenith\.Extensions\.Consul：服务治理、配置中心封装

- Zenith\.Extensions\.Rabbitmq：事件总线、消息可靠投递、幂等、死信队列封装

- Zenith\.Extensions\.Redis：分布式缓存、分布式锁、限流、热点数据

- Zenith\.Extensions\.Elasticsearch：结构化日志、链路追踪、日志归集

## 4\. 业务微服务边界规划（对标老项目四大业务域、全新DDD重构）

对标原有 AgPayPlus 四大后端核心服务：`agpay-manager-api`、`agpay-agent-api`、`agpay-merchant-api`、`agpay-payment-api`。**完全沿用成熟业务边界，不乱拆分、不新增冗余模块**，在此基础上基于 \.NET10 \+ DDD 从零重构，解决老项目架构陈旧、代码耦合、无领域建模的问题。

- **ManagerService 平台管理域（对标 agpay\-manager\-api）**：平台后台配置、支付渠道管理、风控参数、全局订单管理、系统运维配置

- **AgentService 代理分润域（对标 agpay\-agent\-api）**：代理层级管理、分润规则、自动分账、代理结算、收益统计

- **MerchantService 商户域（对标 agpay\-merchant\-api）**：商户入驻、应用接入、费率配置、商户权限、商户数据统计

- **PaymentService 支付核心域（对标 agpay\-payment\-api）**：支付下单、收银台交互、渠道回调处理、订单状态机、退款业务、交易通知、对账清算

## 5\. 核心架构设计理念

### 5\.1 DDD 整洁四层架构（全域统一规范）

所有微服务统一标准四层架构，**领域层零外部依赖、业务与技术彻底解耦**，业务服务无感鉴权、无安全逻辑侵入：

- **Presentation 呈现层**：极简控制器，仅负责请求接收与响应返回

- **Application 应用层**：CQRS 读写分离、业务流程编排、领域事件消费

- **Domain 领域层**：核心业务规则、聚合根、实体、值对象、领域事件、规约（纯业务）

- **Infrastructure 基础设施层**：数据库、缓存、消息队列、第三方渠道、任务调度技术实现

### 5\.2 事件驱动解耦架构

基于 RabbitMQ 实现跨服务事件总线，通过领域事件完成服务解耦，无硬编码依赖。支付成功、退款完成等核心动作发布领域事件，下游服务异步消费，实现高可用、可并行、易扩展的业务流转。

## 6\. 云原生与 DevOps 体系

- **容器化部署**：所有服务独立 Docker 打包、无状态部署、环境一致

- **GitHub Actions 全自动 CI/CD**：提交代码自动构建、单元测试、镜像打包、集群部署

- **K8s 可演进架构**：基于 K3s 稳定运行，支持后续弹性扩缩容、灰度发布、滚动升级、监控运维

## 8\. 支付业务完整 DDD 工程架构（最终落地结构）

### 8\.1 解决方案整体结构

支付全链路业务统一聚合在同一解决方案，高内聚统一规范，基础设施全部独立仓库：

```Plain Text
Zenith.Payment.Platform.sln
├── ManagerService         # 平台管理领域
├── AgentService           # 代理分润领域
├── MerchantService        # 商户领域
└── PaymentService         # 支付核心领域（重点核心）

```

### 8\.2 单服务标准 DDD 四层工程结构（以 PaymentService 为例）

```Plain Text
PaymentService/
├── PaymentService.Presentation          # 呈现层：极简API入口
│   └── Controllers
│
├── PaymentService.Application           # 应用层：CQRS + 业务编排
│   ├── Commands                         # 写操作：下单、支付、退款、状态变更
│   ├── Queries                          # 读操作：订单查询、流水查询、对账统计
│   ├── EventHandlers                    # 领域事件消费处理
│   ├── Dtos
│   └── ApplicationServices
│
├── PaymentService.Domain                # 领域层：纯业务核心（零依赖）
│   ├── Aggregates                       # 聚合根：订单/交易/退款
│   ├── Entities                         # 领域子实体
│   ├── ValueObjects                     # 不可变值对象：金额、渠道、状态、费率
│   ├── DomainEvents                     # 全套支付领域事件
│   ├── Specifications                   # 业务规约校验
│   └── DomainServices                   # 核心业务规则、资金计算、状态机
│
├── PaymentService.Infrastructure        # 基础设施层：所有技术实现
│   ├── Data                             # EF Core、工作单元、仓储
│   ├── Caching                          # Redis 缓存、分布式锁
│   ├── MessageBus                       # RabbitMQ 事件总线
│   ├── ExternalChannel                  # 第三方支付渠道对接
│   ├── Configuration                    # Consul 配置与服务注册
│   ├── Logging                          # ES 日志归集
│   └── JobIntegration                   # Quartz 定时任务对接
│
├── PaymentService.Shared                # 公共通用层
├── PaymentService.Tests                 # 领域单元测试
├── Dockerfile
└── appsettings.json

```

### 8\.3 DDD 架构核心亮点

- 纯正领域驱动，而非数据表驱动，保证金融业务严谨性

- CQRS 读写分离，解决支付高并发下单与高频查询性能冲突

- 事件驱动架构，服务完全解耦、可扩展、可追溯

- 领域层无任何技术依赖，业务逻辑高内聚、可测试、易迭代

- 网关统一鉴权，业务服务纯粹专注业务，架构更优雅安全

## 9\. 项目最终定位（一句话总结）

**Zenith Pay Platform 是参考 AgPayPlus 成熟聚合支付业务模型，基于 \.NET 10、DDD、微服务、云原生从零全新设计实现的企业级支付架构实践项目。项目采用生产级 YARP 网关无状态验签安全架构，搭配 OpenIddict 令牌签发中心、Consul 服务治理、Quartz\.NET 任务调度与自研 Zenith\.Extensions 组件底座，完整落地现代 \.NET 企业级分布式支付系统的领域建模、安全设计与云原生生产级规范。**
