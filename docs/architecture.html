Layer Component & Service Overview

1. Frontend Layer (External Traffic Entrypoint)

Client applications for merchants, operation staff and channel agents. Only HTTPS external requests allowed, no direct access to internal private network resources.

  • Desktop Backoffice: Daily reconciliation, rate configuration, channel management
  • Mobile & H5 Cashier: End-user payment order creation, order status inquiry
  • Third-party Payment Gateways: WeChat/Alipay/Bank asynchronous Webhook callbacks

2. Gateway Layer (Single Public Network Boundary)

All external traffic converges here for security validation, request routing and traffic control. All backend microservices are isolated from public internet.

  • Nginx: L4 load balancing, SSL certificate offloading, static asset forwarding
  • YARP .NET10 Gateway: JWT authentication, dynamic service discovery, circuit breaker & rate limiting, full distributed log tracing. Communicates with internal microservices via standard RESTful HTTP over private network.

3. Application Layer (Core Business Microservices)

Domain-Driven Design separated business domains, CQRS pattern for read/write separation handling merchant, transaction and revenue-sharing core logic.

  • IdentityService: OpenIddict distributed centralized authentication & fine-grained permission control
  • SchedulerService: Quartz.NET distributed task runner for auto reconciliation, refund execution, expired order closing
  • Four domain microservices: Operation Platform, Channel Agent, Merchant Console, Payment Gateway

4. Infrastructure Layer (Distributed Middleware Foundation)

Shared middleware & common components to decouple business logic from distributed capability implementation.

  • Consul: Service registration, health check & distributed KV configuration center
  • RabbitMQ: Asynchronous event queue for traffic peak shaving, system decoupling & transactional messaging
  • Shared utilities: Monitoring, email notification, container runtime, service governance & centralized logging

5. Storage Layer (Persistence, Cache & Log Data Warehouse)

  • SQL Server: ACID transactional storage for core orders, merchant profiles & split settlement records
  • Redis: Distributed lock, hot data cache, user session & rate limit counters
  • ELK Stack: Full API request log & exception log storage, fast search for incident troubleshooting

6. Supporting Infrastructure Layer (Cloud Runtime Base)

Underlying compute, network & security resources for container cluster runtime and private network isolation protection.

7. CI/CD DevOps Pipeline (Automated Build & Deployment)

Fully automated workflow from source code commit to production cluster deployment, supports one-click rollback for version recovery.

End-to-End Payment Request Workflow

Complete client-to-storage request flow for a standard merchant payment order creation:

Step 1 Client Initiates External HTTPS Request

Merchant H5/Desktop Backoffice sends order creation request over public HTTPS, traffic hits public Nginx cluster ingress.

Step 2 Nginx Load Balancing Forwarding

Nginx completes SSL decryption and distributes inbound traffic to healthy YARP gateway instances by weight rules.

Step 3 Unified Preprocessing on YARP Gateway

1. Validate JWT session token & API access permissions; 2. Verify request signature to prevent tampering; 3. Pull up-to-date healthy service list from Consul; 4. Enforce rate limit & circuit breaker protection for downstream services; 5. Generate global TraceId for full request tracing logging.

Step 4 Internal RESTful HTTP Call to Payment Microservice

Gateway sends RESTful HTTP requests to PaymentGateway service via K8s private ClusterIP network, all business services isolated from public internet.

Step 5 Business Service Read/Write Processing Logic

1. Fetch merchant rate configuration from Redis cache; 2. Construct order domain entity; 3. Persist new order record into SQL Server with local database transaction; 4. Publish "Pending Payment" domain event to RabbitMQ message queue.

Step 6 Asynchronous Middleware Consumption & Log Persistence

RabbitMQ consumer workers send payment notification emails and pre-process daily reconciliation data asynchronously; Filebeat collects all request logs and ships to ELK storage cluster.

Step 7 Synchronous Response Returns Back to Client

Payment checkout link & unique order ID propagates back through the full request chain to frontend client, completing synchronous request lifecycle.

Step 8 Asynchronous Channel Callback Finalization (Post Payment)

After end-user completes payment, WeChat/Bank server sends asynchronous Webhook callback to gateway. Gateway re-authenticates callback signature, updates order payment status, generates split settlement data and pushes user payment notifications.

Fault Tolerance & Resilience Mechanisms

  • Unhealthy Service Eviction: Consul health check removes offline instances, YARP automatically routes traffic to healthy service nodes.
  • Traffic Surge Protection: YARP rate limiting + RabbitMQ peak traffic buffering to avoid database overload.
  • Message Loss Prevention: RabbitMQ persistent queue + dead letter queue with scheduled retry compensation tasks.
  • Distributed Troubleshooting: Full request traceability via TraceId cross-search in ELK log storage.
